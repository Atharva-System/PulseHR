# PulseHR AI Platform (IntentBot) - Detailed Project Documentation

This document provides a highly detailed, component-level outline of the **PulseHR AI (IntentBot)** platform architecture. The system is designed as a Multi-Agent AI HR assistant built on LangGraph, integrated with a React SPA frontend and a FastAPI backend using PostgreSQL.

---

## 1. Monorepo Organization & Directory Structure
The architecture is split into two main roots:
- **/frontend**: A React 19 Single Page Application.
- **/hr-ai-platform**: A complex Python backend containing both REST API services and LangGraph-based AI Orchestration.

## 2. In-Depth Database Schema (`hr-ai-platform/db/`)
The system uses **PostgreSQL** configured via SQLAlchemy (`db/models.py`). Let's examine the core tables that map internal workflow:

1. **`users` (UserModel)**: Stores `user`, `hr`, and `higher_authority` identities. Features fields for `role`, `is_active`, and a `receive_notifications` flag (used for SLA routing).
2. **`conversations` (ConversationModel)**: The lifeblood of the stateless chat API. Every AI request/response is logged with a tracing parameter (`trace_id`), intent, and severity. This allows the backend to dynamically pull the last 5 logs for context.
3. **`complaints` (ComplaintModel)**: Specifically designed to log structured analytics when the `employee_complaint` agent is triggered (emotion, severity, target ticket ID).
4. **`tickets` (TicketModel)**: Generated by the Complaint Agent. Has standard HR support fields (`status`, `severity`, `assignee`, `sla_deadline`, `sla_breached`).
5. **`ticket_comments`**: Allows internal HR and higher authority members to communicate secretly about a ticket without alerting the user instantly.
6. **`policies` (PolicyModel)**: Stores the company policies. These are queried dynamically by the `Policy Agent` to answer questions in a grounded way.
7. **`messages` & `feedback`**: Internal staff messaging tables and employee post-resolution feedback tables for analytics.
8. **`audit_log`**: Logs major escalations and system events for review.

*Auto-Migrations*: The system does light auto-migration directly inside `app/main.py`'s `lifespan` event (e.g., executing raw ALTER TABLE scripts). This is critical for agents trying to fix minor schema changes.

## 3. High-Level AI Orchestration Flow (`hr-ai-platform/orchestrator/`)
The AI brain is constructed via **LangGraph**, where each node acts as a functional state mutator.

### 3.1 State Definition (`HRState`)
Every interaction passes an `HRState` dictionary across the graph containing items like:
- `user_id`, `message`
- `conversation_history`
- Metadata like `trace_id`, `intent`, `severity`, and `agent_used`.

### 3.2 Graph Flow (`graph.py`)
1. **START** -> **Router (`classify_intent`)**: Uses structured output LLMs to assign an `intent` (e.g., `employee_complaint`, `leave_request`, `payroll_query`, `policy_question`).
2. **Dispatcher (`dispatch`)**: Examines the `intent`. It checks if that specific agent is toggled ON (saved dynamically, exposed via `api/routes/agents.py`). If active, memory points it to the Agent Subclass. If inactive, routes to `default_agent`.
3. **Target Agent execution**: Each agent processes its logic, alters `HRState` (assigns response text), and concludes.
4. -> **END**.

## 4. Deep-Dive: The AI Agents (`hr-ai-platform/agents/`)

### 4.1 Complaint Agent (`agents/complaint/`)
The most heavily structured subgraph in the repository. It performs a multi-turn HR intake interview instead of immediately fulfilling the request.
- **Node - Classify**: Identifies type, emotion, severity.
- **Node - Safety**: Intercepts requests denoting self-harm or danger and bumps severity to CRITICAL immediately.
- **Node - Policy & History**: Checks prior complaint volumes and relevant rules.
- **Completeness Loop**: LLM evaluates if it has enough info (Who, What, When).
  - If **GATHERING**: Asks a targeted follow-up. Halts graph, returns to User.
  - If **CONFIRMING**: Summarizes and requests explicit authorization to file. 
  - If **COMPLETE**: Generates a final summary, executes `escalate` (creates an HR ticket in the DB), and emails the staff.

### 4.2 Leave, Payroll, & Policy Agents
- **Leave / Payroll Agents**: Designed to parse queries and hit external APIs / Dummy DB records for balance/salary inquiries. 
- **Policy Agent**: Grounded exclusively in RAG. It scans the `policies` table in PostgreSQL, embeds inputs, and strictly answers based only on official definitions.

## 5. Backend REST API (`hr-ai-platform/api/`)
Endpoints expose data primarily for the React SPA. Everything is protected by JWT verification using dependables (`get_current_user`).

- **`/api/chat`**: Standard stateless endpoint for the AI Orchestrator. 
- **`/api/auth`**: JWT Bearer token generation.
- **`/api/admin` & `/api/users`**: RBAC controls for role manipulation, creating new HR accounts.
- **`/api/agents`**: A toggle feature that allows HR to disable specific AI bots gracefully. (e.g. Turn off payroll bot during processing week).
- **`/api/reports`**: Endpoint for HR to aggregate JSON statistics across tickets, severities, SLA breaches.
- **`/api/policies`**: Standard CRUD for the HR team to add new active policies for semantic search.

### SLAs and Escalations
Located in `escalation/sla_checker.py`. Runs via an `asyncio.create_task` loop invoked from `main.py`. This daemon wakes up every X seconds (configurable via `.env`) and parses unresolved `TicketModel` entries past their `sla_deadline`, triggering an `sla_breached = True` flag and dispatching SMTP emails to authorities.

## 6. Detailed Frontend Structure (`frontend/`)
The frontend heavily relies on React 19 functional patterns, Vite for fast HMR compilation, and Tailwind V4.

### 6.1 State & Client Architecture
- **Context API (`src/contexts/AuthContext.tsx`)**: Manages the persistent JWT session state using generic browser LocalStorage. Exposes global variables of `currentUser` containing Role data.
- **API Wrapper (`src/api/client.ts` & `services.ts`)**: `axios` instance heavily integrated with request interceptors to auto-inject the `Authorization: Bearer <token>` into headers.

### 6.2 Layout and Security Context
- **`ProtectedRoute.tsx`**: A crucial wrapper taking an `allowedRoles` array prop. If a `user` attempts to hit an `hr` route, this strictly intercepts and drops them back to their authorized UI.
- **`DashboardLayout.tsx`**: Specialized responsive shell containing an App-bar and side-navigation configured conditionally for `hr` and `higher_authority` identities.

### 6.3 Specialized Pages Flow
- **User Dashboard (`/chat`, `/my-tickets`)**: A highly restrictive view. Chat integrates seamlessly, injecting styled "Cards" into the chat flow if an AI agent returns an "Action" string (like indicating a ticket is created). 
- **HR Triage (`/hr/tickets`, `/hr/chats`)**: Interfaces to manually observe the AI system. Using `ChatViewerPage`, HR reps can directly see the log flow between AI and user. They can read intent vectors, emotions, and assigned trace IDs.
- **Admin Control (`/admin/users`, `/admin/agents`)**: Allows configuration of the LangGraph runtime settings visually.

## 7. Configuration & Environment (`.env`)
Backend uses `.env` files dynamically resolved through `config.py` using `pydantic-settings`. 
Important properties:
- `DATABASE_URL`: PostgreSQL connection logic.
- `JWT_SECRET_KEY`: System-wide authentication secret.
- `SMTP_*`: Credentials necessary for the Complaint Agent's escalation paths.

To interact with this project efficiently, remember that modifications in `hr-ai-platform/orchestrator` immediately impact user chat, while structural logic for what agents are allowed to accomplish is hardcoded into `hr-ai-platform/agents/<name>/graph.py`.
